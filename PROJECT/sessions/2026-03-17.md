### Bugs resueltos (en orden cronológico)

#### 1. ZAP API key incorrecta — raíz de todos los problemas de conectividad
**Síntoma:** ZAP devolvía `"API key incorrect or not supplied: changeme"` para todas las peticiones.
**Causa raíz:** El `.env` tenía `ZAP_API_KEY=cambia-esto-por-una-clave-zap` (placeholder del `.env.example` sin rellenar). Docker Compose expandía la variable al valor del `.env`, no al default `changeme`.
**Fix:** Cambiado `ZAP_API_KEY=cambia-esto-por-una-clave-zap` → `ZAP_API_KEY=changeme` en `.env`.
**Archivos:** `.env`

#### 2. ZAP OOM crash al 34% durante active scan
**Síntoma:** ZAP llegaba siempre al 34% y crasheaba con uso de ~9GB de RAM.
**Causa:** Scan activo contra la raíz de Juice Shop (`/`) abre cientos de hilos nativos JVM. El `-Xmx` no limita off-heap (threads, buffers). Con el app completo, ZAP se queda sin memoria.
**Fix:**
1. Añadido `mem_limit: 3g` al servicio ZAP en docker-compose.yml
2. Añadidos flags: `-config ascan.threadPerHost=2 -config ascan.maxRuleDurationInMins=5 -config ascan.maxScanDurationInMins=60`
3. Cambiar target del scan: `/rest/` en lugar de raíz `/` (scope reducido, menos hilos)
**Archivos:** `docker-compose.yml`

#### 3. `trigger_active_scan` con `--wait` falla por timeout de OpenClaw Exec
**Síntoma:** El agente obtenía `"Cannot connect to ZAP"` pero `docker exec agent-analyst python3 trigger_scan.py --url ...` funcionaba perfectamente.
**Causa:** OpenClaw's `Exec` tool tiene un timeout interno más corto que los 10-15 minutos que tarda el scan activo. El proceso se cortaba y el agente interpretaba el output incompleto como error de conexión.
**Fix:** Actualizado `SKILL.md` para que el agente siempre lance sin `--wait` y haga polling manual del progreso.
**Archivos:** `pentester/skills/trigger_active_scan/SKILL.md`

#### 4. SKILL.md con instrucción obsoleta sobre spider previo
**Síntoma:** El SKILL.md decía al agente que ejecutara `spider_endpoint` antes del active scan, pero esa skill está desactivada en el Analyst.
**Fix:** Actualizado SKILL.md para reflejar que el script hace el spider automáticamente.
**Archivos:** `pentester/skills/trigger_active_scan/SKILL.md`

### 2. Sistema de delegación entre agentes ✅ — skill `delegate_to_agent`

**Problema:** El Supervisor no tenía mecanismo para enviar tareas a los demás agentes.
**Solución:** OpenClaw expone `POST /hooks/agent` en cada agente para recibir tareas vía HTTP.

**Archivos creados:**
- `pentester/skills/delegate_to_agent/SKILL.md` — documentación del flujo completo de orquestación
- `pentester/skills/delegate_to_agent/scripts/delegate.py` — POST a `/hooks/agent` con `Authorization: Bearer {HOOKS_TOKEN}`

**Configuración de hooks (3 bugs seguidos hasta resolverlo):**

| Error | Causa | Fix |
|---|---|---|
| `404 /hooks/agent` | Hooks desactivados por defecto en OpenClaw | Añadir `"hooks": {"enabled": true, "token": "..."}` al config JSON |
| `Unrecognized key: "hooks"` (arranca en crash loop) | `hooks` dentro de `gateway` — clave inválida | Mover `hooks` al nivel raíz del JSON |
| `hooks.token must not match gateway auth token` | `HOOKS_TOKEN=pentester` == `OPENCLAW_GATEWAY_TOKEN=pentester` | Cambiar HOOKS_TOKEN a `hooks-secret` |

**Archivos modificados:**
- `pentester/config/explorer.json` — `hooks` añadido a nivel raíz con `token: hooks-secret`
- `pentester/config/analyst.json` — ídem
- `pentester/config/exploiter.json` — ídem
- `pentester/skills/delegate_to_agent/scripts/delegate.py` — token cambiado de `OPENCLAW_GATEWAY_TOKEN` a `HOOKS_TOKEN`
- `.env` — añadido `HOOKS_TOKEN=hooks-secret`
- `docker-compose.yml` — `HOOKS_TOKEN` propagado a todos los servicios

### 4. Filtrado por target en ChromaDB — `--url-filter` ✅

**Problema:** `query_attack_surface` devolvía findings de todos los targets mezclados (Juice Shop + PortSwigger labs). El informe final listaba vulnerabilidades de Juice Shop con la URL de un lab de PortSwigger.
**Causa:** ChromaDB hace búsqueda semántica sin filtrar por URL. Todos los findings de todos los pentests conviven en la misma colección.
**Fix:** Añadido `--url-filter STR` a `query.py` — post-filtra resultados por substring en el campo `url` de los metadatos (ChromaDB no soporta `$contains` nativo en metadata).

**Archivos modificados:**
- `pentester/skills/query_attack_surface/scripts/query.py` — nueva función `apply_url_filter()`, nuevo argumento `--url-filter`
- `pentester/skills/query_attack_surface/SKILL.md` — documentado el nuevo parámetro
- `pentester/agents/supervisor/SOUL.md` — todas las consultas de ChromaDB actualizadas a `--url-filter {dominio_del_target}`

### 5. Skill `save_report` — informes persistentes en disco ✅

**Fix:** Nueva skill que guarda el informe en Markdown en `/app/reports/` (mapeado a `./reports/` en el host).

**Archivos creados:**

- `docker-compose.yml` — variable `REPORTS_DIR=/app/reports` añadida
- `pentester/config/supervisor.json` — `save_report` habilitada
- `pentester/config/openclaw.json` — `save_report` habilitada
- Directorio `reports/` creado con `.gitkeep`

### 6. Skill `clear_target_data` — limpiar datos antes de re-auditar ✅

**Problema:** Al re-auditar el mismo target, los findings anteriores en ChromaDB contaminan los nuevos resultados.
**Fix:** Nueva skill que elimina todos los findings de ChromaDB cuya URL contenga el dominio indicado, y opcionalmente los archivos de informe en `./reports/`.

**Archivos creados:**
- `pentester/skills/clear_target_data/SKILL.md` — documenta `--dry-run`, `--delete-reports`
- `pentester/skills/clear_target_data/scripts/clear_target.py`

**Uso:**
```bash
# Desde el contenedor (los datos están en el volumen Docker)
docker exec agent-supervisor python3 /app/pentester/skills/clear_target_data/scripts/clear_target.py \
  --domain "0ad500d703bf68d98025.web-security-academy.net" \
  --delete-reports --dry-run
```

**Archivos modificados:**
- `pentester/config/supervisor.json` — `clear_target_data` habilitada
- `pentester/config/openclaw.json` — `clear_target_data` habilitada
